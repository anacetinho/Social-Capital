# Requirements Specification - Social Capital CRM

## 1. System Overview

### 1.1 Project Description
The Social Capital CRM is a comprehensive personal relationship management application that enables users to track, analyze, and optimize their professional and personal networks. The system provides AI-powered insights for strategic networking and relationship maintenance.

### 1.2 Purpose
- Enable users to systematically manage their personal and professional relationships
- Support multi-degree relationship visualization and pathfinding
- Facilitate better relationship outcomes through data-driven insights

### 1.3 Scope
- Dockerized web application running on localhost:7000 with multi-user support
- Comprehensive relationship scoring and analytics
- Interactive network visualization
- Mobile-responsive design with offline capabilities

## 2. Functional Requirements

### 2.1 Authentication & User Management

#### 2.1.1 User Registration
- **FR-001**: System SHALL provide user registration with email and password
- **FR-002**: System SHALL validate email format and password strength
- **FR-003**: System SHALL prevent duplicate user registrations
- **FR-004**: System SHALL redirect users to dashboard after successful registration

#### 2.1.2 User Authentication
- **FR-005**: System SHALL authenticate users with email and password
- **FR-006**: System SHALL use JWT tokens for session management
- **FR-007**: System SHALL provide demo account access (socialcapital@example.com / 123456)
- **FR-008**: System SHALL support user logout functionality

#### 2.1.3 Multi-User Support
- **FR-009**: System SHALL isolate user data completely
- **FR-010**: System SHALL support concurrent user sessions
- **FR-011**: System SHALL maintain user-specific settings and preferences

### 2.2 People Management

#### 2.2.1 Person Creation
- **FR-012**: System SHALL allow users to create person records with name, contact info, and biographical data
- **FR-013**: System SHALL support profile picture upload with automatic resizing (400x400px)
- **FR-014**: System SHALL assign unique IDs to each person
- **FR-015**: System SHALL validate required fields and data formats

#### 2.2.2 Person Information
- **FR-016**: System SHALL store comprehensive person data including:
  - Basic info: name, email, phone, LinkedIn URL
  - Biographical data: birthday, job title, company, net worth estimates
  - Contact preferences: communication methods, response patterns

#### 2.2.3 Person Profile Management
- **FR-017**: System SHALL provide detailed person profile pages
- **FR-018**: System SHALL allow editing of all person fields
- **FR-019**: System SHALL support person deletion with confirmation
- **FR-020**: System SHALL cascade delete related relationships, events, and favors

### 2.3 Relationship Management

#### 2.3.1 Relationship Creation
- **FR-021**: System SHALL support bidirectional relationship creation between any two people
- **FR-022**: System SHALL provide kinship type selection (family, friend, colleague, etc.)
- **FR-023**: System SHALL allow relationship strength scoring (1-10 scale)
- **FR-024**: System SHALL prevent duplicate relationships
- **FR-025**: System SHALL support relationship verification status

#### 2.3.2 Network Visualization
- **FR-026**: System SHALL provide interactive network graph visualization using D3.js
- **FR-029**: System SHALL support filtering by relationship type 
- **FR-030**: System SHALL provide zoom, pan, and hover interactions
Also network nodes proximity should be based on relationship type. immediate family (1- close), extended family (2- medium), business and aquaintances (3- far).

#### 2.3.3 Connection Pathfinding
- **FR-031**: System SHALL calculate shortest paths between any two people
- **FR-032**: System SHALL use BFS algorithm for optimal connection routing
- **FR-033**: System SHALL highlight connection paths on network graph
- **FR-034**: System SHALL analyze multi-degree connections (up to 3 degrees)
- **FR-035**: System SHALL calculate path strength based on relationship scores

### 2.4 Event Management

#### 2.4.1 Event Creation
- **FR-036**: System SHALL allow users to log events with title, description, date, and location
- **FR-037**: System SHALL support multiple participants from user's network
- **FR-038**: System SHALL provide interaction quality rating (1-5 scale)
- **FR-039**: System SHALL support intimacy level rating (1-5 scale)
- **FR-040**: System SHALL categorize events by type (meeting, call, message, favor, conflict, celebration)

#### 2.4.2 Event Management
- **FR-041**: System SHALL provide event listing with filtering options
- **FR-042**: System SHALL allow event editing and deletion
- **FR-043**: System SHALL display events chronologically
- **FR-044**: System SHALL show event history on person profiles
- **FR-045**: System SHALL use events in relationship scoring calculations

### 2.5 Favor Tracking

#### 2.5.1 Favor Recording
- **FR-046**: System SHALL track favors with giver, receiver, and description
- **FR-047**: System SHALL categorize favors (professional, personal, financial, social)
- **FR-048**: System SHALL allow value estimation and impact scoring
- **FR-049**: System SHALL track reciprocity status
- **FR-050**: System SHALL monitor favor balance for each relationship

#### 2.5.2 Favor Management
- **FR-051**: System SHALL provide favor listing with filtering options
- **FR-052**: System SHALL allow favor editing and deletion
- **FR-053**: System SHALL display favor exchange patterns
- **FR-054**: System SHALL show favor history on person profiles
- **FR-055**: System SHALL incorporate favors into relationship scoring
